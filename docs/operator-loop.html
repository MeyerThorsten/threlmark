Plan my day: a queue that argues its case

Priority × status weight, plus risk boosts — the same primitives as the Insights risk register, so the plan and the register can never disagree.

Scoring

score = priority × status-weight
  + 12 overdue  + 6 due ≤ 3 days
  + 8 stalled handoff  + 4 × items it blocks
  + 5 stale in Development

→ every boost is also a human reason: "overdue since 2026-06-01", "blocks 2 other items", "finishing beats starting".

→ a security risk outranked everything. That's the point.

Webhooks & rules: one file, whole contract

// ~/.threlmark/automations.json
{
  "webhooks": [
    { "url": "https://example.test/hook", "events": ["item.done", "report.received"] }
  ],
  "rules": [
    { "on": "item.moved", "toLane": "development", "addLabels": ["wip"] }
  ]
}
EventsGuarantees
item.created · item.moved · item.done
report.received · handoff.recorded		 fire-and-forget (3s timeout) — slow endpoints never slow the board · every attempt logged to events.log (JSONL) · automations can never break a mutation

Snapshots & the weekly digest

⬇ Snapshot

The whole board — four lanes, priorities, labels, initiative bars — as one self-contained HTML file. No scripts, no external requests; verified by test. Email it, attach it, host it. GET /api/projects/:id/snapshot

Week in review

Shipped (with outcomes + agent attribution), started, new items, current risks, forecast, open initiatives — reconstructed from item history, no bookkeeping. ?format=html|md · ?days=30 · ?save=1 drops dated files into ~/.threlmark/digests/ for cron.

Doctor & backup: the local-first promise, kept

$ npm run doctor
✖ ERROR  projects/readiness/items/178109…json
         projectId "Readiness" does not match owning project "readiness"
12 errors · 0 warnings · 0 info        # exit code 1 → cron/CI-friendly

$ npm run backup
✓ ~/.threlmark/backups/threlmark-2026-06-11T04-43-12.tgz (411 KB)
1 backup(s) kept
It earned its keep on day one. The doctor's first run against the real store found 12 items whose projectId had been written with the wrong casing by an external tool — invisible at runtime (reads normalize it away), wrong on disk. Found and fixed the same hour. Restores are just tar -xzf: the store is plain files.
