Hosting Threlmark

Two separate things can be "hosted" on Threlmark.com, with very different requirements: the public website (easy, static) and the Threlmark app itself (local-first — a public, multi-user service is a different product).

1 · The public website (site/)

The files in site/ are plain static HTML/CSS with no build step and no third-party requests (no font CDN, no analytics) — deliberately, so the site is DSGVO-clean and runs on any web host.

Deploy to all-inkl.com

  • Upload the contents of site/ to your web root via FTP/SFTP or the KAS file manager.
  • index.html is the homepage; privacy.html, terms.html, contact.html, impressum.html are linked in every page's footer.
  • No server config needed. (Optional: an .htaccess to force HTTPS.)
✅ Already deployed and live at threlmark.com. The "Open the app" button links to /app — see below for what should live there.

2 · Can Threlmark.com offer a hosted version of the tool?

Short answer: not as-is on all-inkl shared hosting, and not as a single shared instance. Threlmark is intentionally local-first: one JSON store under ~/.threlmark, no authentication, no multi-tenancy. On a public URL unchanged, every visitor would share and edit one roadmap store.

Also: all-inkl shared hosting serves PHP/static. Next.js needs a Node.js runtime for its API routes (which read/write the disk store), which the standard packages don't provide. The marketing site runs fine there; the app needs a Node host.

Option A — Static demo (read-only) easiest · fits all-inkl

Ship a read-only, sample-data demo at /app: bundle seeded JSON and route writes to the browser's localStorage (exactly like the original Roadmap Lab). Full look and feel; nothing persists server-side; no auth.

Effort: small — add a demo flag that serves bundled JSON and sends writes to localStorage.

Option B — Personal hosted instance recommended for your own use

Run the real app on a small Node host behind a password so only you use it.

  • Host on a VPS (Hetzner), Fly.io, or a Node container; point app.threlmark.com at it.
  • Set THRELMARK_DATA_DIR to a persistent, backed-up volume.
  • Add an auth gate (HTTP Basic via reverse proxy, or a single-password middleware).

Effort: small–medium. No core code changes — just deployment + an auth layer.

Option C — True multi-tenant SaaS a separate build

A public "sign up and get your own roadmaps" service — a real product change, not a deploy:

  • Accounts + auth (the app currently has none).
  • Per-tenant data isolation — a data dir per account (the contract namespaces cleanly under a root) or a database adapter behind the store interface.
  • A Node host with persistent, backed-up storage, plus rate limits, billing, and GDPR data-subject tooling.

Effort: large. The src/lib/*/store.ts boundary is the natural seam to swap disk for per-tenant storage.

Recommendation

  • Now: the static site is live; add Option A (read-only demo) at /app so "Open the app" shows the tool.
  • For your own online use: Option B on a small Node host.
  • Only build Option C if you want to offer accounts to other people.

Keeping the deployed copy current

  • Static site: a deploy step (FTP/SFTP sync, or a GitHub Action with an FTP action) that uploads site/ on each change.
  • App (Option B): deploy from git on the Node host (git pull && npm ci && npm run build && pm2 restart), or a container image rebuilt on push.
⚠️ Keep deploy credentials out of any public git repository. Site content and secrets should never be committed to a public remote.
