Threlmark

A local-first hub that manages all your projects in one place and runs a ranked, kanban-style roadmap for each — with the data stored as plain JSON on disk so your other tools can read a project's roadmap and write suggestions back into it.

Overview

Threlmark generalizes a single-product roadmap (the localStorage Roadmap Lab kanban) into a multi-project hub. Every app you build becomes a project with its own scored kanban; a cross-project Portfolio ranks what matters across all of them; an Inbox surfaces suggestions other tools drop in; and handoff briefs turn selected cards into Claude/Codex implementation prompts.

Built with Next.js (App Router) + TypeScript. No database, no cloud, no accounts. Disk JSON under ~/.threlmark is the single source of truth.

System architecture

The UI (server components) and external tools both reach the same store functions — one through React, one through the REST API. Stores read and write JSON files atomically. There is no in-memory state, so everything survives a restart.

Persistence discipline

Every entity is a JSON file that is its own source of truth. Writes go to a temp file then rename over the target (atomic on one filesystem), so a crash mid-write can never truncate the file. Updates read the current file, spread-merge the patch, preserve id/createdAt, bump updatedAt, and preserve unknown keys for forward-compatibility.

The shared data contract

Disk is the source of truth. External tools may read/write these exact shapes. The schema is versioned (schemaVersion) with lazy, per-file migration.

suggestions/<sugId>.json — what a tool writes

Only source and title are required; everything else defaults on read. targetProjectId makes accept promote the item into a different project.

Disk layout

Per-item files (not one monolithic roadmap.json) let external tools write concurrently without clobbering a shared array. board.json self-heals on read: any item present in items/ but missing from its status lane is appended; orphan ids are dropped.

~/.threlmark/
├─ threlmark.json                 # manifest { schemaVersion }
├─ links.json                     # cross-project dependency graph
├─ projects/
│  └─ <projectId>/
│     ├─ project.json             # project metadata
│     ├─ board.json               # lane ordering (self-healing)
│     ├─ items/<itemId>.json      # card + transitions[] + handoff ✦
│     ├─ suggestions/<id>.json    # external drop zone = the Inbox ✦
│     ├─ suggestions/.dismissed/  # accepted/dismissed (audit)
│     ├─ handoffs/<id>.json       # recorded agent handoffs
│     ├─ reports/<file>.json      # agent report-back (ingested on read)
│     └─ ROADMAP.md               # regenerated human mirror
├─ shared/items/<itemId>.json     # shared across projects
└─ archive/projects/<projectId>/  # archived (still readable)
✦ External tools only ever need to write items/ and suggestions/. They never touch board.json or links.json — those self-reconcile.

Scoring & priority

Each item is scored 1–5 on four axes. Priority is a single computed number, reused verbatim from the original Roadmap Lab so imported cards rank identically.

priority = max(0, round(impact·3 + evidence·2 + fit·2 − effort·1.5))

Axis weights

Worked example

A card scored impact 5, evidence 5, fit 5, effort 4:

5·3 + 5·2 + 5·2 − 4·1.5
= 15 + 10 + 10 − 6
= 29

Impact is weighted heaviest; effort is the only axis that reduces priority.

Portfolio ranking

Across all active projects, items are ranked by a status-weighted score, so in-flight work rises to the top of the whole portfolio. Items that block others get a small bottleneck boost.

score = priority · statusWeight   (+ 0.1 · blockedCount · priority)

Flow & agent throughput

The board is a flow visualization, not just status tracking. Every item's append-only transitions[] history makes the four flow metrics computable. This is the modern-Kanban evolution: limit work-in-progress, measure flow, improve incrementally — without the enterprise apparatus (no SLAs, no deterministic delivery dates).

MetricDefinition
Work-item ageTime in the current lane (now − last transition). Past a per-lane threshold (dev 7d · ranked 21d · idea 60d) the card is flagged stale.
Cycle timeEntering Development → reaching Done (falls back to lead time if it skipped Development).
ThroughputItems reaching Done per ISO week (8-week window), per project and across the portfolio.
WIP & limitCount per lane; a lane over its wipLimits cap is flagged.

Agent flow — the 2026 twist

Most work now ships via AI coding agents, so a handoff is a first-class flow event. Recording one (Handoff tab → Generate & mark handed off, or POST …/handoff with record:true) stamps each item with handoff:{handoffId, agent, at} and writes a record under handoffs/. The Flow view then measures:

  • Brief → shipped — Done time − handoff time.
  • Agent throughput — handed-off items reaching Done per week, grouped by agent.
  • Stalled briefs — handed off, still not Done after 7 days.

Report-back closes the loop

The brief carries a reporting protocol: Claude/Codex report started / done / blocked / failed (with a summary + the verification commands they ran) — via POST …/items/<id>/report or by dropping a JSON into reports/ (ingested on read). A done report auto-moves the card to Done and the board shows a live toast. No manual step: rank → hand off → agent builds & reports → Done.

Cross-project model

Four operations connect projects. All use the global address "<projectId>/<itemId>".

OperationWhat happens on disk
MoveItem file rewritten into project B, removed from A; both boards fixed; link addresses repointed.
Link / dependAn edge appended to links.json; drives the portfolio dependency view + bottleneck boost.
ShareCanonical card written to shared/items/; each source becomes a thin sharedRef pointer; duplicates links recorded.

Key workflows

Import

Extract the defaults = [...] array from roadmap.html, parse it tolerantly with JSON5 (it's JS literals, not JSON), write one item file per card keeping original ids. Re-import updates in place.

Suggestion → Inbox → accept

A tool drops a JSON into suggestions/. The Inbox polls every few seconds. Accept creates an item with source set and files the suggestion under .dismissed/.

Handoff

Select cards → generate a file-scoped Claude/Codex brief with acceptance checkboxes and verification commands, or queue text / JSON. Copy or download.

Local-first principles

Yours, on disk

No cloud, no accounts, no secrets. Everything is JSON under ~/.threlmark — inspect it, back it up, sync it with whatever you like.

Interoperable

The disk layout is the API. Any tool can read a roadmap or drop a suggestion with a single file write, and Threlmark reconciles on read.

Crash-safe

Atomic temp-then-rename writes mean a half-written file can never replace a good one.

Forward-compatible

Unknown fields survive a round-trip; schemaVersion enables lazy migration without a big-bang upgrade.

Ecosystem: Threlmark + IdeaClyst

The disk contract isn't just for Threlmark — other local-first tools read and write it. The first is IdeaClyst, an idea engine (Claude↔Codex "council" + web scouting). Threlmark is the system of record for roadmaps; IdeaClyst is an idea source that contributes to them.

Reads roadmaps

IdeaClyst's src/lib/threlmark/ layer reads a project's items + board read-only and computes a gap map (which categories/lanes are thin) — matching Threlmark's per-item-file format and priority exactly.

Writes suggestions

It drops scored Features / Spin-offs / Services into suggestions/ (atomic write), grounded in live web research, tagged source:ideaclyst with provenance — over disk or the REST API, with disk fallback.

You stay in control

Suggestions surface in Threlmark's Inbox. Accept → ranked item; hand off → agent builds & reports back → Done. The loop feeds the next analysis: shrinking gaps change what IdeaClyst proposes.
